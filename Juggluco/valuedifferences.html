Glucose value differences between FreeStyle Libre 2 apps


There are several ways to get continues glucose monitoring from a FreeStyle Libre 2 sensor.

As with FreeStyle Libre 1, you can place Bluetooth sensor on top of a FreeStyle Libre 2 sensor, like MiaoMiao, DiaBox and Blucon. New with FreeStyle Libre 2 are apps that directly get glucose values from the sensor like the patched Librelink app+ xdrip and Juggluco. (On some people Diabox gives the impression to do the same, but that is not true. Diabox can scan Freestyle libre 2 sensors and displays the scanned value in the same way as Juggluco displays values received via Bluetooth. The value in Diabox value is also present with Bluetooth turned off and a constant value stays in the android status bar forever. In contrast, Juggluco refreshes this value very minute with the glucose value received via Bluetooth and removes the values if for three minutes it didn't receive a new glucose value.)

The LibreLink app scans a sensor, it doesn't display the raw values it receives from the sensor. The app processes the data received from the sensor to come to the glucose values shown to the user and used for alarms. FreeStyle Libre 1 apps like Glimp and Liapp, did give their own interpretation of the raw sensor readouts. None of the apps who are used with FreeStyle Libre 2 sensors does so. All work like the patched version of Librelink: they use part of the Librelink app to interpret the raw sensor readouts.

The value the alternative apps give is different because they use an old version of Librelink to interpret the data. This version is nearly always the one also used by the patched librelink app, namely the German version of Librelink 2.3.0.

How they incorporate this app differs. The so called patched version of the Librelink app is the most open about this. It changes some code in Librelink 2.3.0 so that it sends data to xdrip (and optionally turns of verifications and communication with Abbott). What is less well-known is that it also contains an unmodified version of the Librelink app in assets/original.apk of its apk, there by doubling its size. As android installs an app the APK stays in the device. When the patched Librelink is run it extracts assets/original.apk out of the apk and saves it in

/data/data/com.freestylelibre.app.de/no_backup/original.apk. Thus the the installed version of the patched Librelink contains three times Librelink 2.3.0.

DiaBox is not very different. In res/raw/original_apk it contains the German Librelink 2.3.0 and as it is run it is saved in /data/data/com.outshineiot.diabox/filesbase111.apk

It also contains libDataProcessing.so libMathRuntimeChecks.so libSecureKeyBoxJava.so out of Librelink 2.3.0.

LinkBluCon and the non-us Tomato app from Miaomiao both contain contain the German version of Librelink 2.3.0 in res/raw/original_apk and lib/arm64-v8a/libDataProcessing.so and the directory tree in java/com/abbottdiabetescare/flashglucose/sensorabstractionservice.

The US 14 days version of the Tomato app for miaomiao contains the US version of Librelink 2.2.1 in res/raw/original_apk.

Out of legal reasons, the Juggluco package at Google play doesn't contain anything from Abbott, but when run for the first time it takes libDataProcessing.so out of the installed Librelink or asks the user to download it if Librelink is not present (or only the old, patched, Librelink is present). Juggluco is the only that doesn't use any of the Librelink Java code.Because Juggluco uses libDataProcessing.so from the new versions of Abbotts Librelink, it gives the same glucose values as the new versions of Librelink. And there is a large change between Librelink 2.3.0 and Librelink 2.4.2. There after the values have not changed anymore.
